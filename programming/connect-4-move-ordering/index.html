In order to achieve the highest possible efficiency of the alpha-beta algorithm, good move ordering is necessary. In contrast to the Minimax algorithm, which has to examine all nodes of the search tree, the Alpha-Beta algorithm can prune large parts of the tree if the best moves are tried out early on. The reason for this is that the opponents always try to maximize their own results and minimize the opponent’s results. The more the current results come closer to the game theoretical values, the more subtrees can be cut off, as they no longer contribute to the determination of the exact values. Cutoffs in lower levels of the search tree cut off larger subtrees. Since the computing effort is limited due to the smaller number of nodes in these levels, sophisticated algorithms for move ordering are conceivable there, which increase the computing time per node, but in return achieve higher cutoff rates. If this is implemented in a reasonable way, significant runtime improvements are possible. Therefore, you should first try to further improve move ordering before making technical optimizations and other changes to the program. This offers much more potential. Our agent’s alpha-beta search is divided into two stages. In the first step, it is possible to implement runtime-intensive procedures, since the number of searched nodes is comparatively small. The second stage, on the other hand, is very static to minimize the computing time per node. In the following, the techniques for move ordering are discussed in more detail for both stages.

  Note: The related Java framework for Connect-4 is available on GitHub and can be downloaded from: http://github.com/MarkusThill/Connect-Four.

Simple Move Ordering

Generally speaking, it can be said that plys in the central columns of the board are usually preferable. Placing them in central columns increases the likelihood of creating threats that are critical to the outcome of the game. In the second stage of our search, this simple sorting is still used because it does not require any dynamic elements and can therefore be implemented very efficiently. To date, no further measures have been successful in improving the second stage of the move ordering. This is mainly due to the fact that cutoffs in lower levels of the search tree do not save enough nodes to compensate for the additional effort. For the first stage, however, further refinements of the above-mentioned approach are possible in order to achieve additional runtime reductions. Central fields can be involved in more chains of four than those lying at the edge. For example, in addition to the column, the row of a cell can also be used to estimate the expected quality of a ply. For instance, a ply in cell c3 is usually superior to a ply in cell d6, since the former theoretically allows a larger number of chains of four to be laid. The figure below shows how many chains of length four can be put through each single cell of the board. The source code to compute these values is listed in the appendix.

A further aspect to assess the quality of a ply can be the neighboring fields of a cell. Cells with many adjacent opponent stones are often worthless for a player, even if they are centrally located. Plys into such cells should therefore be avoided by both players if possible. The following approach can be used to improve the move ordering: To do this, match all four-chains in which the respective cell is included with the opponent’s stones and devaluate the corresponding move with each match (since the opponent’s stones make it impossible for the current player to completely occupy this four-chain). This is repeated in each node, but can be done very efficiently using bitwise AND operations and the corresponding masks. Furthermore, this type of move ordering is only used in nodes close to the root.

Preference for Moves which generate Threats

However, the probably most important optimization approach has been omitted so far: Especially in the early and middle phase of a game it is important for the players to build up threats. If one of the two players succeeds in doing so, the other may be forced to make a move and win the game. However, not all threats of a player are always helpful to finally defeat the opponent. Both players must pursue different strategies to generate threats. Above all, the position (usually the row is decisive) and the arrangement of one or more threats determine how a position is to be judged. Victor Allis has elaborated in detail on this topic in his master thesis. However, the rules set up are so complex and abstract that it is very difficult to transfer them to move ordering. Since move ordering is not always able to provide the optimal move sequence for a node in all cases, some generalizations that cover as many situations as possible are sufficient here:

  1. Generally, the player who has more threats in odd rows (odd threats) wins.
  2. The player with the first move (1st player) usually benefits only from odd threats.
  3. Even threats in particular are only of use to the 2nd player. However, as soon as the 1st player has at least one odd threat, even threats are useless.

Considering these general statements, it is possible to adjust the train sorting as follows: For the 1st player, moves that generate an odd threat are preferentially examined, regardless of the position of the placed stone. This also applies to the 2nd player, with the exception that even threats in addition are taken into account. If it is possible to create several threats in the current node, moves in the middle columns are to be treated with priority. In many cases, one of the two players must be put under Zugzwang (avoiding an immediate defeat) in order for the opponent to force a suitable threat. Moves of this kind are also detected by the move ordering and tried out before the remaining moves. As the search for threats is comparatively complex, this can only be done at higher levels of the alpha-beta search. Nevertheless, the runtime advantages are considerable, the average search time of the agent is halved due to this measure.

For reasons of efficiency and to avoid errors, the corresponding source code was generated automatically, which results in a very long code (more than 2000 lines). The generating program is listed in the appendix.

Summary

Depending on the depth of search, the move ordering can be constructed more or less time-consuming. High cutoff rates are possible with the help of dynamic move ordering. Due to the increased effort involved in sorting, however, it is only possible to use it in lower levels of the search tree. In many cases, static move ordering does not deliver as good results as the dynamic one, but minimizes the computing time per node. Therefore, static move ordering is particularly suitable for higher search tree levels. Due to the two-step structure of the Alpha-Beta search, both methods can be used. In the first stage, dynamic move ordering is used, which sorts the moves according to the following priority list: Before the actual move ordering begins, unnecessary moves are sorted out. If a position is symmetrical, the reflection at the middle axis therefore provides the same position, it is sufficient to examine only the first four columns.

  1. Moves that generate a (non-immediate) threat have the highest priority. Only odd threats are relevant to the 1st player.
  2. moves that put the opponent under direct Zugzwang and then allow the creation of their own suitable threat have a high priority.
  3. The remaining moves are then examined. Centrally located cells receive a higher priority than those at the edge, taking into account the neighboring opposing stones.
  4. Here, the search sorts moves to the back, which move under one’s own threats. This happens considering the fact that the opponent neutralizes this threat next turn and consequently renders it useless. Here too, only odd threats are of importance to the 1st player.

The second stage of the Alpha-Beta search uses the trivial, mostly static move ordering. Starting from the middle column, the moves are executed in the direction of the outer columns. Only moves under one’s own threat are avoided and not investigated until the end.
