For Connect-4, a perfectly playing agent must be able to fully examine the game tree (in extreme cases 42 moves), but the search effort increases exponentially with the search depth, so that in many respects a high development effort is necessary to achieve this goal. This post and the following are intended to explain some of the procedures and techniques used to enable such an agent. In this blog post, we will particularly look at a basic tree search algorithm: the alpha-beta search.

  Note: The related Java framework for Connect-4 is available on GitHub and can be downloaded from: http://github.com/MarkusThill/Connect-Four.

The Alpha-Beta Search

The Alpha-Beta search is a tree-search algorithm based on the classic Minimax algorithm, which has been optimized in some parts. While a simple Minimax search examines all nodes of the game tree, using two variables (alpha and beta) in the alpha-beta search may cut off large subtrees during the search. However, the number and extent of cutoffs depend on several factors such as the quality of the move ordering. The basic idea behind the algorithm is as follows: Since the opponents, due to their maximizing or minimizing behavior, avoid certain game courses if they are already aware of better alternatives, the investigation of whole subtrees is unnecessary, since certain plies would not be considered anyway. As already mentioned, two variables play a central role: Alpha describes the lower bound of the maximizing player and corresponds to the value that this player will reach at least at the moment. The variable Beta, on the other hand, contains the upper bound of the minimizing player and indicates the maximum value that is reached by the minimizing player. Due to the recursive nature of the alpha-beta algorithm, the alpha and beta values in each node of the tree can be changed and compared with the current values, so that cutoffs are not limited to the root node, but can be made at any point in the tree. From the maximizing player’s point of view, the following steps are taken to process a tree node: First of all, the search determines all possible legal moves that are then successively tested. If the search returns a value greater than or equal to Beta for one of the following states, the current node is immediately exited and Beta is returned. This is a fail-high. The returned value corresponds to a lower bound of the actual game-theoretical value of the position. A fail-low occurs in a node when Alpha has not yet been adjusted a single time and therefore no move could improve the existing result. When leaving the node, the routine returns Alpha, the upper bound for the exact value of the position. Only values between alpha and beta are exact, in all other cases only bounds are determined, because the precise values for the outcome of the game are not relevant. In his introduction to chess programming, Bruce Moreland describes the above facts very succinctly and comprehensively:

“A fail-high indicates that the search found something that was “too good”. What this means is that the opponent has some way, already found by the search, of avoiding this position, so you have to assume that they’ll do this. If they can avoid this position, there is no longer any need to search successors, since this position won’t happen. A fail-low indicates that this position was not good enough for us. We will not reach this position, because we have some other means of reaching a position that is better. We will not make the move that allowed the opponent to put us in this position. Minimizing nodes can be inspected in the same way as the aforementioned.”
