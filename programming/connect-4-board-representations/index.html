The representation of the board is an elementary component of the agent. Since very large parts of the program require this data structure as a basis, special care must be taken in the specification of this data structure. This applies in particular to tree-search procedures such as the alpha-beta search, in which a large number of operations are performed on this data structure. Probably the simplest representation of the board is achieved in most games by using a one- or two-dimensional array. For Connect-4, for example, a 7x6 integer matrix could be used. This approach is also recommended for less complex games such as Tic Tac Toe, since efficiency is not particularly important and the representation of the board simplifies programming in many parts.

  Note: The related Java framework for Connect-4 is available on GitHub and can be downloaded from: http://github.com/MarkusThill/Connect-Four.

For more complex games such as Connect-4, Nine-Mens-Morris, Checkers or Chess, such a representation is often inconvenient. In these games, great importance must be attached to the efficiency of the algorithms and the underlying data structures. Reducing the Minimax or Alpha-Beta algorithm to the most basic operations quickly reveals that field manipulations (performing / undoing moves) and the identification of terminal states (win / loss or draw) play a central role. These operations all take place on the data structure of the game board. For the game Connect-4, the search for wins or losses (closed four-member chains), which takes place in practically all nodes of the game tree, can become an agent’s bottleneck. Especially this search for closed chains of four is comparatively complex when using arrays, especially if they have to be executed many millions of times in a certain time. For this reason, so-called bit boards were used for our alpha-beta agent, which have some basic advantages over simple arrays. Above all, the time-critical search for chains of four can be carried out efficiently with the help of this representation.

Bit Boards

In chess programming bit boards (often in combination with simple arrays) have been used for a long time to represent the board. In particular, the number of 64 fields in the chessboard makes the use of bit boards interesting, since most systems now have 64-bit architectures and 64-bit variables can be processed directly with a CPU instruction. Games like Othello or Checkers with 64 fields could also benefit from this. Basically, bit boards are simple variables, but they are considered at the bit level. Usually, the individual bits of a variable correspond to a position on the board. By setting or deleting the bits, individual pieces can be placed or removed at the corresponding position. In order to save some time during the tree search, it makes sense to use predefined masks. The runtime advantages that result from this are not to be underestimated. Since bit boards do not contain any information about the type of piece, only pieces of the same type can be held in a bit board. For example, when playing chess, all the white player’s pawn could be kept in a 64-bit variable, but not bishops which needed their own bit board. Connect-4 knows only two types of tiles, which means that two bit boards (one for each player) are needed. Each bit board requires 42 bits. It is therefore advisable to use 64-bit variables (e. g. of type Double or Long in Java) which use the least significant 42 bit (as in this project). Two 64-bit variables are therefore sufficient to completely describe a Connect-4 position. Placing a tile can be done with the help of a bitwise OR operation, deleting with a bitwise AND operation. Only the current ply is relevant for the recognition of a winning position. As a result, only a local search for chains of four in the surroundings of the occupied cell is necessary. Based on some bitwise AND operations performed on the player’s bit board and the corresponding bit masks (one for each possible string of four) any quads can be determined quickly. In our work, the winning search is implemented in a similar way, with the exception that the inverted bit boards are used here. However, the number of AND operations depends on the placement of the tile. Generally speaking, it can be said that more operations are required for central fields than for external ones, since in principle more potential chains of four are possible in the central fields.

The implementation of the data structures and the important operations can be found in the class CFour/src/c4/ConnectFour, which is also listed in the appendix of this post.

Advantages and Disadvantages of Bit Boards

Altogether, a number of advantages of bitboards over arrays can be identified (related to the game Connect-4; not all advantages are necessarily transferable to other games):

  • Many operations can be carried out more efficiently, since they can include several fields of the board at the same time. Examples are the recognition of winning positions and draws, the mirroring of playing boards along the middle column and the determination of threats.
  • Copying the board can be done much faster because only two 64-bit variables represent the board.
  • Reduced memory requirements associated with transposition tables.

However, the use of bit boards can have some disadvantages that should not be neglected:

  • High programming effort: In some cases, parts of the source code were also generated with other programs.
  • Poor readability of the source code.
  • Maintenance of the program is difficult.
  • Fault-prone and tedious troubleshooting.
  • Very long program code: All in all, nearly 7000 lines of source code were needed to realize this agent.
